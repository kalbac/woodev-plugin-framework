# План рефакторинга фреймворка (черновик)

> Это рабочий черновик-брейншторм, **не строгая спека**. После согласования из него
> родится отдельная спека для реализации (через `superpowers:writing-plans`).
> Цель документа — зафиксировать намерения и открытые вопросы так, чтобы будущий
> агент не путался, когда мы приступим к реализации.

---

## 1. Цели рефакторинга

- Фреймворк должен брать на себя всю основную часть для построения на его базе плагинов WooDev.
- В плагинах остаётся **только доменная логика**.
- Фреймворк должен поддерживать плагины:
  - для чистого WordPress,
  - для WooCommerce,
  - для Easy Digital Downloads (EDD),
  - и иметь возможность расширяться дальше (другие платформы).
- Стремимся к DI настолько, насколько это возможно в рамках WordPress, и в целом к SOLID.
- Исполняемые классы фреймворка — это **каркас**, использующий контракты.
  Классы плагинов их расширяют.
- Стремимся к реактивному UI в админке там, где это целесообразно и не стоит
  дорого при разработке (например, таблица заказов или основных настроек).

---

## 2. Общая идея

Сейчас весь фреймворк ориентирован только под плагины для WooCommerce.
Нужно сделать так, чтобы базовый класс **не имел зависимостей** от WooCommerce, EDD и т.д.

Целевая иерархия классов:

```php
class Woodev_Plugin {}                                      // чистый WordPress
class Woodev_Woocommerce_Plugin extends Woodev_Plugin {}    // плагины WooCommerce
class Woodev_EDD_Plugin extends Woodev_Plugin {}            // плагины Easy Digital Downloads
```

Принцип распределения:

- Все базовые функции/хелперы/классы подключаются внутри `Woodev_Plugin`.
- Всё, что нужно только в окружении WooCommerce, — внутри `Woodev_Woocommerce_Plugin`.
- Всё, что нужно только в окружении EDD, — внутри `Woodev_EDD_Plugin`.
- И так далее для будущих платформ.

### Стратегия выполнения

- Рефакторинг покроет большую часть кода фреймворка → делаем его **отдельной git-веткой**,
  чтобы оригинал всегда был под рукой для сверки.
- Перед стартом: при необходимости делаем ревью того, что есть, затем
  детальный брейншторм, разбиваем всё на задачи и подзадачи, для каждой
  пишем план и/или спеку.
- По сути мы **делаем новый фреймворк** — от старого мало что останется.
  Делаем современно, используя лучшие практики.
- Не боимся что-то сломать в плагинах, которые используют старые версии
  фреймворка: их потом всё равно будем переписывать под новый.

---

## 3. Иерархия классов

Распределение существующей функциональности по новой иерархии:

- **`Woodev_Plugin`** (чистый WordPress) — базовые хелперы, общие сервисы.
- **`Woodev_Woocommerce_Plugin extends Woodev_Plugin`** — всё, что зависит от WC.
- **`Woodev_EDD_Plugin extends Woodev_Plugin`** — всё, что зависит от EDD.

### Подтверждённые перемещения

- Класс `Woodev_Lifecycle` гарантированно переезжает в `Woodev_Plugin`
  (он не должен зависеть от WC).
- Модуль `api/` тоже должен гарантированно уйти под окружение `Woodev_Plugin`
  (см. раздел 4.3).

## 4. Модули

Внутри фреймворка есть папки с кодом, отвечающие за конкретный функционал —
называем их **модулями**.

### 4.1 Модуль оплаты — `payment-gateway`

В целом работает хорошо. Менять особо ничего не нужно. При необходимости
просто адаптировать код этого модуля под новые условия (новую иерархию классов).

### 4.2 Модуль доставки — `shipping-method`

Сейчас в стадии разработки. Очень важный блок, который нужно будет реализовать
полноценно — это отдельная большая работа, которой нужно уделить больше внимания.

Так как основная часть плагинов WooDev — это плагины доставки, **этот модуль
должен получиться идеальным и максимально универсальным** для переиспользования.

**Архитектурный референс:** модуль оплаты (`payment-gateway`) —
у него примерно та архитектура, к которой мы хотим прийти и в shipping.

**Референсные плагины** (для понимания того, как сейчас работают плагины
доставки WooDev, какие паттерны встречаются):

- `D:\Projects\wordpress\woocommerce-edostavka`
- `D:\Projects\wordpress\woodev-russian-post`
- `D:\Projects\wordpress\woocommerce-yandex-delivery` — особенно стоит взять
  как референс **карту ПВЗ**: именно в этом плагине она самая стабильная.

### 4.3 Модуль API — `api`

Должен гарантированно уйти под окружение `Woodev_Plugin` (т.е. перестать

## 6. Открытые вопросы

Эти решения нужно принять **до** того, как из этого черновика родится спека.

1. **Судьба `bootstrap.php`.**
   Сейчас `bootstrap.php` выполняет свою роль очень хорошо, но он как будто
   становится лишней прослойкой. Нужно решить, что с ним делать в новой архитектуре.

2. **Способ объявления типа плагина.**
   Сейчас, чтобы пометить плагин как платёжный шлюз, мы передаём параметр
   `is_payment_gateway` — это неудобный и нестабильный подход. Возможно, стоит
   в entry-файле плагина сразу наследоваться от класса, соответствующего его типу
   (`Woodev_Woocommerce_Plugin`, будущий `Woodev_Payment_Gateway_Plugin` и т.д.).
   Нужно зафиксировать конечный механизм.
