# План рефакторинга фреймворка (черновик)

> Рабочий черновик-брейншторм, **не строгая спека**. После согласования из него
> родится отдельная спека для реализации (через `superpowers:writing-plans`).
> Цель документа — зафиксировать намерения и открытые вопросы так, чтобы будущий
> агент не путался при переходе к реализации.

---

## 1. Цели рефакторинга

| Направление | Формулировка |
|-------------|--------------|
| Роль фреймворка | Берёт на себя всю основную инфраструктуру для плагинов WooDev; в плагинах остаётся **только доменная логика**. |
| Платформы | Поддержка плагинов для чистого WordPress, WooCommerce, Easy Digital Downloads (EDD) и возможность расширения на другие платформы. |
| Архитектура | DI — насколько возможно в WordPress; в целом — SOLID. Исполняемые классы фреймворка — **каркас** на контрактах; плагины их расширяют. |
| Админ UI | Реактивный UI там, где это целесообразно и не дорого в разработке (например, таблица заказов или основных настроек). |

## 2. Проблема и целевая модель

**Сейчас:** фреймворк ориентирован только на плагины WooCommerce.

**Нужно:** базовый класс **без зависимостей** от WooCommerce, EDD и других платформ.

### 2.1 Иерархия классов

```php
class Woodev_Plugin {}                                      // чистый WordPress
class Woodev_Woocommerce_Plugin extends Woodev_Plugin {}    // плагины WooCommerce
class Woodev_EDD_Plugin extends Woodev_Plugin {}            // плагины Easy Digital Downloads
```

### 2.2 Принцип распределения кода

| Уровень | Что подключается |
|---------|------------------|
| `Woodev_Plugin` | Базовые функции, хелперы, общие сервисы |
| `Woodev_Woocommerce_Plugin` | Всё, что нужно только в окружении WooCommerce |
| `Woodev_EDD_Plugin` | Всё, что нужно только в окружении EDD |
| *(будущие платформы)* | По тому же принципу |

### 2.3 Подтверждённые перемещения

- `Woodev_Lifecycle` → в `Woodev_Plugin` (не должен зависеть от WC).
- Модуль `api/` → под `Woodev_Plugin` (см. [§ 4.3](#43-модуль-api--api)).

### 2.4 Стратегия выполнения

- Рефакторинг затронет большую часть кода → **отдельная git-ветка**, оригинал остаётся для сверки.
- Перед стартом: при необходимости — ревью текущего кода; затем детальный брейншторм, разбивка на задачи и подзадачи, для каждой — план и/или спека.
- По сути — **новый фреймворк**; от старого мало что останется. Делаем современно, с лучшими практиками.
- Поломки в плагинах на старых версиях фреймворка не страшны: их всё равно перепишем под новый.

### 3.4 Модуль лицензий — `licensing`

#### Плагины без лицензии

- Флаг `is_need_license`, значение по умолчанию `true`; переопределяется в плагине.
- При `false` на странице лицензий (Woodev → Лицензии):
  - либо не показываем поле ввода лицензии,
  - либо показываем, что лицензия не требуется.
- Все проверки актуальности лицензии игнорируются; **возможность получать обновления сохраняется**.

#### UI страницы лицензий

- Сделать максимально интерактивной и современной (сейчас — обычные input + submit).
- **Обратная совместимость** в плагинах должна сохраниться.

#### Вебхуки

Система вебхуков — см. [§ 3.4.1](#341-встроенные-вебхуки).

### 3.4.1 Встроенные вебхуки

Два сценария (пока два):

1. **Просроченная лицензия.** Плагин продолжает использоваться после окончания срока, пользователь игнорирует уведомления о продлении → возможность **отключить или удалить** плагин на сайте пользователя через запрос к вебхуку. Операция должна быть безопасной: выполнить её может только мы.
2. **Диагностика для `woodev_support`.** Для будущих AI-агентов поддержки — диагностика сайта и плагина без доступа к админке: агент запрашивает вебхук, получает детали, анализирует.

### 3.5 Модуль «Упаковщик» — `box-packer`

| | |
|---|---|
| **Статус** | Универсальный, может работать отдельно от фреймворка; есть недостатки. |

**Задачи:**

- Сейчас привязан к WooCommerce. Теоретически вне WC не применяется, но при универсальном фреймворке модуль должен работать и на других платформах → нужен **качественный wrapper**.
- Главная проблема: не умеет формировать «виртуальные» коробки из габаритов товаров с **минимальным итоговым размером** коробки. Подробнее — [§ 3.5.1](#351-виртуальная-коробка-минимальный-размер).

### 3.5.1 Виртуальная коробка: минимальный размер

**Текущий алгоритм:** берёт наибольшие стороны по осям, оставшиеся — складывает.

**Пример, где всё ок (2 товара):**

| Товар | Габариты |
|-------|----------|
| 1 | 10 × 10 × 5 |
| 2 | 15 × 10 × 20 |

Результат: 20 × 15 × 15 (20 — max, 15 — второй max, 5+10=15 — сумма остальных).

**Проблема (3+ товара):** к примеру выше добавить третий товар 10 × 10 × 5:

- Текущий алгоритм: **20 × 15 × 20** (сумма «оставшихся» 5+10+5=20 сравнялась с одной из наибольших сторон).
- При 10 товарах: **20 × 15 × 55**.

**Суть:** модуль не пытается уместить товары так, чтобы итоговая коробка занимала **минимально возможное** пространство.

## 5. Открытые вопросы

Решить **до** превращения черновика в спеку:

1. **Судьба `bootstrap.php`.** Сейчас выполняет роль хорошо, но в новой архитектуре выглядит лишней прослойкой. Нужно решить, что с ним делать.
2. **Способ объявления типа плагина.** Сейчас тип платёжного шлюза задаётся параметром `is_payment_gateway` — неудобно и нестабильно. Вариант: в entry-файле плагина сразу наследоваться от класса нужного типа (`Woodev_Woocommerce_Plugin`, будущий `Woodev_Payment_Gateway_Plugin` и т.д.). Нужно зафиксировать конечный механизм.

## 6. Заметки и предпочтения

### UI

При реактивной части админки — **встроенный React WordPress и WooCommerce**, не отдельный ReactJS в проекте.

### Работа с агентами

На этапе планирования и написания спеков ожидается формат **дискуссии**, а не только строгое выполнение задачи: агент инициативен, предлагает идеи, оспаривает предложения оператора.
