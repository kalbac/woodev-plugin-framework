## 1. Основная цель модуля

Модуль должен полностью освободить плагины доставки от DRY-операций — в самих плагинах остаётся **только доменная логика**.
По итогу все плагины доставки должны работать **одинаково и предсказуемо**. Пользователь, разобравшийся с плагином CDEK,
не должен гадать «а как это делается?» в плагине Яндекс Доставки. Всё стандартизировано и универсально.

Принцип выноса: если есть хоть один повторяющийся паттерн настроек/поведения — мы выносим его переиспользование на уровень
абстракции. В самом плагине он объявляется либо через `support`, либо через `FEATURE`, либо фреймворк сам определяет его
по типу плагина (доставка, оплата и т.д.).

> 📎 Из ревью кода: гипотеза «всё повторяется» — **подтвердилась**. По девяти из пятнадцати измерений три плагина делают
> практически одно и то же, расходясь только в деталях API конкретного перевозчика. Ниже — матрица «общее vs домен»,
> которая и есть скелет всего модуля.

## 3. Существующие паттерны (нарратив оператора)

Все существующие плагины имеют как минимум 2 типа доставки: до ПВЗ и курьером до двери.

Для ПВЗ всегда используется карта для выбора ПВЗ. Все плагины должны иметь одинаковый UX/UI при использовании карты.
Параметры карты всегда определяются в основных настройках плагина. JS карты должен быть спроектирован так, что все значения
передаются в PHP — он должен быть **универсальным**. На стороне плагина мы JS не пишем (кроме доменного).

Нужно учесть, что есть плагины, у которых карта ПВЗ рендерится не плагином, а через **iframe** (так сделано у Почты РФ —
у Почты нет возможности получить отделения для конкретного населённого пункта, поэтому используется готовый виджет в модальном
окне). При рефакторинге Почты нужно рассмотреть вариант, чтобы рендерилась наша карта.

> 📎 Из ревью кода: iframe — **не уникальный случай Почты**. У Яндекса тоже есть iframe-режим (официальный `ndd-widget`),
> и одновременно «своя» карта на Я.Картах, и режим простого `<select>`. У CDEK — выбор между виджетом на Я.Картах и нативным
> CDEK-виджетом. То есть «режим рендера карты» — это **настройка** (`map_type` / `widget_type`), и фреймворк должен
> поддерживать минимум: (а) нашу карту на Я.Картах, (б) сторонний iframe-виджет перевозчика, (в) `<select>`-фолбэк.
> Общими у всех также являются: позиция кнопки выбора ПВЗ (`under-methods` vs `inline-method`), модалка на Backbone,
> поток «выбор в карте → AJAX → сессия → meta заказа».

Все плагины доставки так или иначе управляют полями формы заказа, но каждый делает это по-своему, что создаёт конфликты.
Это нужно организовать централизованно. Пример: поле «Населённый пункт» — CDEK вешает на него выпадающий список через SelectWoo,
а Яндекс (у которого нет своего списка городов) вешает авто-подсказки от Dadata.

> 📎 Из ревью кода: конфликт уже «решён» костылём — см. §8 (Dadata). Это подтверждает необходимость централизованного
> реестра полей.

## 4. Контракты плагина доставки

Идея: плагин **декларирует** свои возможности, а фреймворк по декларации включает нужные подсистемы. Объявление — через
`support`/`FEATURE` или автоматически по типу плагина.

> 📎 Из ревью кода: реальные «оси», по которым плагины различаются и которые стоит сделать декларируемыми:
> - тип аутентификации (oauth / basic+token / token) и нужен ли отдельный канал для трекинга;
> - форма методов (один класс с тарифами vs несколько классов);
> - режим карты ПВЗ (наша / iframe / select);
> - использование Dadata и на каких полях;
> - поддержка НПП / страховки / объявленной ценности;
> - поддержка печати накладной/ШК и «вызова курьера»;
> - источник статусов (webhook / cron / оба);
> - мульти-склад (origin).

✅ **РЕШЕНО (s32).** Механизм декларации зафиксирован, два вида:
> 1. **Boolean support-флаги** (в духе WC `supports`, на методе/плагине): `supports_pickup_points`,
>    `supports_tracking`, `supports_cod`, `supports_insurance`, `supports_declared_value`,
>    `supports_label_printing`, `supports_courier_call`.
> 2. **Config-контракты** (не boolean): auth (тип + опц. канал трекинга), map mode
>    (our_map/iframe/select), status source (webhook/cron/both), field-needs (Dadata через реестр полей §8),
>    форма методов (1-класс-N-тарифов vs N-классов).
>
> Тип плагина — авто по `extends` (s27 убрал `capabilities`). **Финальные имена/дефолты флагов — при
> миграции первого плагина.**

## 5. Аутентификация

Все плагины доставки (но не обязательно) имеют обязательные поля аутентификации (подключение к API), но у всех они разные.
Нужен контракт, который определяет, **какая именно** аутентификация используется и используется ли вообще. На странице
настроек это должно определяться через `support`/`FEATURE`.

> 📎 Из ревью кода — три реальные схемы:
> - **CDEK:** OAuth-подобный обмен `client_id`+`client_secret` → bearer-токен. Токен кэшируется в transient
>   (`woocommerce_edostavka_access_token`) и **рефрешится по запросу** перед вызовами. Креды в `woocommerce_edostavka_settings`.
> - **Почта:** base64(`account_name`:`account_password`) + явный `account_token`. **Отдельные креды** `tracking_login`/
>   `tracking_password` для SOAP-трекинга. Всё — в `woocommerce_wc_russian_post_settings`. Плюс GUID-регистрация
>   (`wc_russian_post_guid_id` / `wc_russian_post_guid_key`) — см. §17.
> - **Яндекс:** голый bearer-`auth_token` + переключатель `environment` (production/test, в тесте — захардкоженный токен).
>
> **Вывод для контракта:** контракт аутентификации должен описывать не только «тип», но и **жизненный цикл**:
> хранение, шифрование, рефреш/истечение, инвалидацию при смене логина/пароля, и **возможность отдельного канала
> креденшелов** (как SOAP-трекинг у Почты).

✅ **РЕШЕНО (s32).** Шифрование БД **не мандатируем** (мы не хранилище секретов). Вместо этого:
> - **`sensitive`-флаг** → маскирование в UI / логах API / REST / экспорте — **всегда**, независимо от источника.
> - **Опц. `constant_name`** на чувствительной опции: если `defined()` — значение берётся из константы
>   wp-config, поле в UI disabled с подсказкой, save его пропускает (не затирает). Секрет вообще не попадает
>   в БД (признано **лучше** шифрования-at-rest). Имя константы задаёт домен, механизм — фреймворк.
> - Дефолт без константы — плейнтекст.

## 6. Расчёт тарифа и упаковка

> 📎 **Новый раздел** — в исходном черновике темы не было, хотя это ядро `WC_Shipping_Method`.
>
> Из ревью кода — у всех трёх плагинов одинаковый скелет:
> 1. `calculate_shipping($package)` → собрать параметры → вызвать API перевозчика → распарсить цену и срок.
> 2. **Кэширование** ответа (CDEK: object-cache по `md5(serialize(params))`, 1ч; Почта: WC transients; Яндекс:
>    `Woodev_Cacheable_API_Base`). Без кэша API перевозчика ляжет от нагрузки на чекауте — это обязательная часть контракта.
> 3. **Наценка/цена** — почти буква-в-букву совпадает: `fee` + `fee_type` (`order` = % от заказа / `shipping` = % от доставки),
>    `free`/`free_cost` (порог бесплатной доставки), `min_cost`/`max_cost`, округление `round_cost` + `round_cost_range`,
>    учёт купонов бесплатной доставки. У CDEK дополнительно `static_price` (фиксированная цена вместо API) и `fee_payments`
>    (наценка только для определённых способов оплаты).
> 4. **Упаковка** через box-packer фреймворка: дефолтные габариты/вес (`default_weight/height/width/length`) как фолбэк,
>    если у товара нет размеров; методы `per_item` / `box_packing` / `single_box`.
>
> **Вывод:** `get_rate_cost()` (наценка/округление/пороги), слой кэширования и интеграция с packer — **переезжают во фреймворк
> целиком**. В плагине остаётся только «собрать параметры запроса для своего API и достать из ответа цену+срок».

✅ **РЕШЕНО (s32).** Фреймворк **не строит фолбэков** (это домен). Дефолт при сбое API — **скрыть метод** +
> лог (если включён режим логирования). Фреймворк даёт **`do_action`-хук**, через который плагин/пользователь
> реализует свой фолбэк, если хочет. **Кэш-как-фолбэк отвергнут** (устаревшая ставка → заказ по тарифу,
> которого на момент экспорта уже нет). Кэш *успешных* ставок для производительности остаётся — это другое.

## 7. ПВЗ и карта

(Нарратив — см. §3.) Карта универсальна, все значения уходят в PHP, плагин JS не пишет.

> 📎 Из ревью кода — общий поток у всех: кнопка «Выбрать ПВЗ» (позиция настраивается) → модалка (Backbone) → выбор точки →
> AJAX → сохранение в WC-сессию (свой data-store) → запись в meta заказа при оформлении. Списки ПВЗ кэшируются (CDEK — неделю).
> Режимы рендера (см. §3): наша карта на Я.Картах / сторонний iframe / `<select>`.
>
> Узкие места, которые стоит заложить в контракт карты:
> - **производительность**: списки ПВЗ бывают на тысячи точек — нужна стратегия загрузки/кластеризации;
> - **безопасность моста JS↔PHP**: раз все значения уходят в PHP через AJAX — единый контракт nonce/прав на эндпоинтах;
> - **мобильная модалка**.

✅ **РЕШЕНО (s32).** Три режима (настройка метода `map_type`): **наша карта на Я.Картах — основной**;
> **iframe — полноправный** (Почту на нашу карту перевести нельзя — нет API отделений по городу; Яндекс/CDEK
> — как вариация); **`<select>` — дешёвый опциональный фолбэк** (делаем, только если тривиально, иначе
> опускаем). **Ключ Я.Карт — фреймворк-конфиг карты (одно место), НЕ §9-сервис.** В контракт карты:
> производительность (тысячи ПВЗ → загрузка/кластеризация), единый nonce/права на AJAX, мобильная модалка.

## 8. Поля формы заказа + состояние (самая больная точка)

С полями формы заказа нужно хорошо поработать — на данный момент это **самая больная точка** всех плагинов доставки. Как минимум
потому, что изначально логика заложена неправильно: после каждого JS-события `checkout_updated` поля полностью перерисовываются
в DOM, и события, которые на них висели, «сиротеют» — приходится заново их навешивать. Получается каша.

> 📎 Из ревью кода — каждый из трёх плагинов **решил эту проблему по-своему**, что и есть источник «каши»:
> - **CDEK:** состояние локации хранится **вне DOM** в собственных `WC_Data` data-stores (user-meta для залогиненных +
>   WC-сессия для гостей); каскад «страна→регион→город» на Backbone-модели.
> - **Почта:** делегирование событий на `body` + **ре-инициализация** Dadata на каждый `updated_checkout`; состояние ПВЗ — в
>   WC-сессии.
> - **Яндекс:** на каждый `updated_checkout` **destroy/recreate** инстансов SelectWoo + hidden-inputs для значений + сессия.
>
> **Корень проблемы** один: обработчики вешаются **прямо на узлы DOM**, которые WooCommerce пересоздаёт. Правильное лечение —
> не привязываться к узлам:
> 1. **состояние полей живёт вне DOM** (единый стор фреймворка), DOM — производное;
> 2. **делегирование событий** на стабильный контейнер вместо прямого биндинга;
> 3. **единый реестр полей**: кто на какое поле что вешает (список/подсказки/кнопка) — централизованно, чтобы плагины
>    не конфликтовали за одно поле.
>
> Это же напрямую связывает данный раздел с §11 (блочный чекаут): блоки используют React-state — то есть «состояние вне DOM»
> там встроено. Если строить слой полей правильно, **одна абстракция закрывает и классику, и блоки**.

✅ **РЕШЕНО (s32).** Целевая модель — **стор состояния вне DOM + реестр полей + делегирование** (не точечная
> починка ре-биндинга). Уточнение формулировки: общее — это **ядро** (стор + реестр + доменная логика: какие
> поля, каскад регион→город→адрес, валидация, AJAX), а рендер — **два тонких адаптера** (классика / блоки)
> поверх одного ядра. «Одна абстракция рендерит и то и то» — неточно; правильно «одно ядро + два адаптера».
> **Валидация (A2, s32):** метод требует ПВЗ, а он не выбран → ядро полей **блокирует оформление** с понятной
> ошибкой (классика + блоки) — часть контракта полей.

## 9. Общие переиспользуемые поля настроек и Dadata

В плагинах часто встречаются одинаковые опции, используемые в разных плагинах. Пример — «Авто-подсказки от Dadata». Нужно
реализовать так, что если один плагин их использует, то второй уже не может (например, показываем уведомление «Опцию X уже
использует плагин Y»).

**Авто-подсказки от Dadata.** Используются во всех трёх плагинах (CDEK, Почта, Яндекс), но без явной согласованности — от какого
именно плагина брать подсказки. У WooCommerce появилась возможность подключать единого провайдера подсказок
(<https://developer.woocommerce.com/docs/features/address-autocomplete/>). Казалось бы, идеально — но есть нюанс: подсказки от
Dadata там работают только с полями «Адрес» и «Почтовый индекс», а нам нужно вешать их и на «Регион», и на «Населённый пункт».
Причём логика каскадная: пока покупатель не указал населённый пункт — он не может выбрать адрес, а адрес подсказывается только
из выбранного населённого пункта.

> 📎 Из ревью кода — **взаимное исключение уже существует, но в виде костылей**:
> - Почта **молча уступает** управление Dadata-полями плагину CDEK, если тот активен (`integration-edostavka.php`).
> - Яндекс **отключает свои Dadata-поля**, если определён `WC_CDEK_SHIPPING_VERSION` (хардкод проверки версии).
>
> То есть текущая модель — «кто костыльнее, тот и владеет». Чище: **Dadata/авто-подсказки — это сервис уровня фреймворка** с
> одной общей конфигурацией (token+secret) и одним UI, не принадлежащий ни одному плагину. Плагины лишь **декларируют** «мне
> нужны подсказки на поле X», а конфликт разрешается на уровне общего сервиса (а не блокировкой второго плагина и не проверкой
> версии конкретного соседа). Каскад «регион→город→адрес» — это логика **одного** сервиса, а не трёх плагинов.
>
> У всех трёх набор тумблеров одинаков: `enable_suggestions_state` / `_city` / `_address` + автозаполнение индекса по адресу +
> детект города по IP. Это прямой кандидат в общий UI.

✅ **РЕШЕНО (s32).** Реестр общих сервисов стартует **только с Dadata** (token+secret, тумблеры
> state/city/address, автозаполнение индекса, geo-IP детект города — внутри этого же сервиса, каскад). Костыли
> (version-check соседа, молчаливая уступка) выпиливаем. **Реестр расширяемый** — новый сервис добавляется
> через общий интерфейс «сервис» (id, схема настроек → своя вкладка, декларация полей, точки подключения), без
> переписывания логики. **Важно:** §9 (реестр *сервисов*) ≠ реестр *полей* §8; сервис застолбляет поле через §8.

## 10. Наложенный платёж, страховка, объявленная ценность

> 📎 **Новый раздел** — в исходном черновике не было, но по коду это реальная сквозная тема (особенно для РФ-рынка):
> - **Почта:** полноценный НПП — `payattr=4` для COD, `insr-value` (сумма страховки), объявленная ценность по позициям.
> - **Яндекс:** `payment_method=card_on_receipt` для НПП, `total_assessed_price` для страховки, **показ комиссии** покупателю
>   при выборе НПП (с тултипом).
> - **CDEK:** сервис `INSURANCE`, объявленная ценность = сумма заказа; явного НПП в этой версии нет.
>
> Тема пересекается с **модулем оплаты**: НПП влияет на итог и на доступность gateway’ев (Яндекс отключает `cod`, если ПВЗ его
> не поддерживает). Нужен контракт «поддерживает ли метод НПП / страховку / объявленную ценность» + хук координации с оплатой.

✅ **РЕШЕНО (s32).** Контракт-флаги `supports_cod` / `supports_insurance` / `supports_declared_value` + **один
> хук координации** «доступные gateway'и при данном методе/ПВЗ» (Яндекс отключает `cod`, если ПВЗ его не
> поддерживает). **Детали расчёта — домен** (комиссия, `payattr=4`, `insr-value`, `total_assessed_price`,
> показ комиссии покупателю).

## 11. Блочный чекаут Gutenberg

Сейчас ни один плагин не поддерживает блочный чекаут и корзину (`WooCommerce Checkout Blocks` / `Cart Blocks`), но всё больше
пользователей этого требуют. Изначально не реализовывали, потому что на старте блоки не давали кастомизировать форму заказа
(а нам критично: к «Населённому пункту» прикрутить выпадающий список (CDEK) или подсказки (все), отрендерить кнопку «Выбрать
ПВЗ» и т.д.).

Нужно разобраться, даёт ли WooCommerce сейчас кастомизировать форму/корзину. Если да — реализовать поддержку блоков **и обратную
совместимость** для классических шорткодов `[woocommerce_checkout]` / `[woocommerce_cart]`.

> 📎 Из ревью кода: см. §8 — если слой полей строится как «состояние вне DOM + реестр полей», то блоки (React-state) и классика
> закрываются **одной** абстракцией. Это аргумент делать §8 и §11 как одну задачу, а не две.

✅ **РЕШЕНО (s32).** **Классика первой, блоки — обязательный фолоу** (ядро §8 проектируем block-ready; блоки
> не должны ломать классику). На карточках товара уже указано «Checkout Blocks не поддерживаются» — не сюрприз.
> Минимум для блоков с первого релиза: поле «город» (с источником значений), поле «регион», **кнопка
> «Выбрать ПВЗ» + модалка карты**, корректная запись в order-meta/сессию. **Референс реализации (обязателен):**
> WC address-autocomplete API — околоидеал, но только для полей Адрес/Индекс (нам нужны Регион/Город), поэтому
> как образец, не переиспользуем as-is.

## 12. Создание отправления, накладная, ШК

> 📎 Из ревью кода — общий flow у всех трёх:
> - экспорт заказа (вручную из админки **и** авто по смене статуса WC), идемпотентность (не экспортировать дважды);
> - получение **трек-номера** + сохранение в meta;
> - печать документов: CDEK — PDF-накладная + ШК-стикер (форматы A4/A5/A6); Яндекс — `print_label` + акт приёма-передачи;
>   Почта — barcode (саму накладную берут в ЛК Почты);
> - доменные действия: CDEK — «вызвать курьера»; Яндекс — выбор оффера (модалка) перед подтверждением.
>
> **Вывод:** общий каркас — стейт-машина отправления (черновик→создано→накладная→…) + генерация/скачивание PDF + флаги
> «скачано». Детали API и форматы документов — домен.

✅ **РЕШЕНО (s32).** Стейт-машина экспорта (отдельно от §13): `not_created → created (remote-id+трек) →
label_ready → handed_over` + `export_failed`. **Авто-экспорт** настраиваемый («создавать при статусе X»,
дефолт ВЫКЛ), ручной всегда доступен, **идемпотентность** по remote-id. **Документы — абстракция «источник
документа»** со стратегиями: `binary_content` (CDEK отдаёт байты PDF → фреймворк оборачивает в download),
`remote_url` (Яндекс даёт ссылку → редирект/прокси), `local_generated` (плагин генерит сам; на будущее). Плагин
реализует `get_document(type)`, фреймворк — download-ответ/имя/флаги. PDF не храним. Доменные действия (вызов
курьера, выбор оффера) — обвязка кнопка→AJAX→доменный хендлер; печать/курьер по support-флагам §4.
**Отмена (A3, v1):** отмена созданного отправления у перевозчика при отмене/возврате заказа (доменное действие
+ обвязка + статус `cancelled`). **Возвраты/реверс-логистика — defer**, но ядро (стейт-машина + источники
документов + трекинг §13) проектируем так, что возврат = «ещё одно отправление в обратную сторону» — добавим
позже дёшево.

## 13. Трекинг и синхронизация статусов

DRAFT (нарратив): отслеживание заказа в админке и на странице покупателя; не все доставщики дают трекинг; изменение статуса
заказа по завершению/отказу/возврату.

> 📎 Из ревью кода — у всех трёх **два механизма сразу**:
> - **push (webhook):** CDEK — через WC legacy API (`ORDER_STATUS`, `PRINT_FORM`, `DOWNLOAD_PHOTO`); Почта — подписанные GUID
>   эндпоинты; Яндекс — REST API.
> - **pull (cron):** опрос статусов по интервалу (CDEK/Яндекс — настраиваемый, Почта — SOAP каждые 12ч), батчами, через
>   `WC()->queue()` / Action Scheduler.
> - **маппинг статусов** перевозчика → статусы WC настраивается (`status_delivered` и т.п.), смена статуса бросает доменные
>   экшены (`..._status_changed_from_{old}_to_{new}`), хранится **история статусов**.
>
> **Вывод:** во фреймворк — диспетчер «push+pull», абстрактный webhook-эндпоинт с верификацией, cron-синхронизатор поверх
> background-job хендлера, конфиг маппинга статусов и стейт-машина с историей. Перевозчик-специфика — только парсинг ответа и
> таблица соответствия статусов.

✅ **РЕШЕНО (s32).** Канонический набор (~9): `pending / created / in_transit / ready_for_pickup / delivered /
> returning / returned / failed / cancelled`. Перевозчик маппит свои статусы в них; **сырой ярлык перевозчика
> сохраняется** для отображения + **полная история**. WC-статус на каждый канонический — настраиваемый
> (`delivered` → WC `completed`). `ready_for_pickup` **трекаем** (мерчант видит «в пункте, ждёт получателя»), но
> **письмо по нему по умолчанию выкл** — перевозчик уведомляет сам. «Статус трекается» ≠ «шлём по нему письмо»
> (письма — §14). **NB:** §12 (стейт-машина *экспорта*) — отдельная от §13 (прогресс *доставки*).

## 14. Уведомления / письма

DRAFT (нарратив): единый контракт уведомлений (заказ отправлен, в пути, доставлен).

> 📎 Из ревью кода: все используют кастомные `WC_Email`, привязанные к смене статуса. CDEK — 3 письма (tracking / delivered /
> not-delivered), Почта — 1 (tracking), Яндекс — 1 (sharing-url). У всех — плейсхолдеры (`{order_number}`, `{tracking_number}`,
> `{tracking_url}`…), кастомизируемый текст, дедупликация по meta-флагу «письмо отправлено».
>
> **Вывод:** общая абстракция «status-driven email» с реестром плейсхолдеров и дедупликацией — во фреймворк; конкретные триггеры
> и шаблоны — домен. Хорошо ложится на «канонические статусы» из §13.

✅ **РЕШЕНО (s32).** Фреймворк даёт базу + реестр плейсхолдеров (`{order_number}`, `{tracking_number}`,
`{tracking_url}`, `{carrier_name}`, `{pickup_point}`, `{delivery_date}`…) + дедуп, **и готовый набор писем**
(каждое toggle + редактируемый richtext-шаблон) на канонических статусах §13: «Передан в доставку»/трек на

`returned`/`failed` (**OFF**); `ready_for_pickup` — без письма. Доменные письма (Яндекс sharing-url) — на той же базе.

## 15. Страница настроек

Это вопрос не столько к модулю доставки, сколько ко **всему фреймворку**: как и где мы рендерим страницу настроек плагина.

Для плагинов оплаты (`payment_gateway`) используется встроенная страница настроек шлюза — как правило, её хватает; иногда нужна
отдельная страница. Для плагинов доставки практически всегда нужна отдельная страница общих настроек. Сейчас во всех плагинах
это наследование `WC_Integration`. Но для WP- или будущих EDD-плагинов эта страница недоступна. Плюс в v2 мы строим страницы
на **React** — значит, нужно сразу продумать схему построения страниц настроек.

Варианты: (а) для доставки по умолчанию строим на `WC_Integration`, только UI переделываем на React; (б) резервируем слот в WP,
где рендерятся все наши страницы настроек независимо от WP/WC/EDD (по умолчанию всегда «как для WP»).

> 📎 Из ревью кода: **сегодня все три плагина — `WC_Integration`** (CDEK, Почта, Яндекс), плюс у Почты есть setup-wizard поверх
> фреймворкового `Woodev_Plugin_Setup_Wizard`. У всех есть **кастомные типы полей** (`warehouses_list`, `pickpoint_list`,
> `cron_update`, `auth_status`) и инстанс-настройки на уровне метода доставки. То есть «реальность» = `WC_Integration` +
> кастомные поля; миграция на React/слот WP должна это воспроизвести.

> ⚠️ Это решение **блокирующее**: от него зависят §16 (страница заказов) и §8/§11 (поля/блоки). Предлагаю вынести его в отдельный
> ADR до старта реализации.

✅ **РЕШЕНО (s32) — вариант (б), ADR-worthy.** Две поверхности:
> 1. **Глобальные настройки (новое):** меню **`Woodev > Настройки`** (в хабе Woodev рядом с Лицензии/Плагины),
>    появляется при ≥1 плагине с настройками. Структура **страница → вкладки → (опц.) подсекции**, правило
>    **1 вкладка = 1 провайдер** (перевозчик ИЛИ фреймворк-сервис типа DaData; мульти-карьер плагин §18 даёт
>    несколько вкладок). Рендер — **один нейтральный React-слот поверх Settings-API через `woodev/v1`** (паттерн
>    визарда/License). Реестр собирает вкладки; схема Settings-API получает 2 уровня группировки (вкладка/секция).
> 2. **Инстанс-настройки метода (без изменений):** остаются в **модалке метода WC Shipping Zones** (тип тарифа,
>    наценка, упаковка) — зональная модель WC.
>
> **Миграция:** плагин декларирует легаси option-ключ (`woocommerce_edostavka_settings`…), Settings-API пишет в
> него; легаси-URL старой страницы → admin-редирект на новую вкладку.

## 16. Страница заказов / админка

DRAFT (нарратив): общий контракт, UI на React, детали заказа в модальном окне, кнопка «создать заказ», фильтры, сортировка,
графики (какая доля заказов приходится на тот или иной плагин доставки — ROI).

> 📎 Из ревью кода: у всех трёх **уже есть** своя страница списка заказов (`WP_List_Table`), метабокс на странице заказа, bulk-
> действия (export / update / print / cancel) и AJAX поверх нонсов. У Яндекса — модалка выбора оффера. То есть контракт админки
> заказов выводится из уже существующего общего знаменателя; на React переносится именно он.
>
> Кандидаты во фреймворк: общая таблица заказов доставки (колонки/фильтры/сортировка), метабокс, набор bulk-действий, слой PDF-
> скачивания, графики/ROI как опциональный модуль.

✅ **РЕШЕНО (s32).** **По одной странице на перевозчика** (дефолт) — иначе каша. Фреймворк даёт переиспользуемые
> блоки (таблица/фильтры/сортировка, метабокс, bulk, модалка, слой печати PDF), каждый перевозчик монтирует свою
> страницу (зеркалит §15: 1 провайдер = 1 страница). **Одна агрегирующая** — только если сам плагин даёт
> мульти-доставку (будущее, не планируется). **ROI/графики — опциональная фича, не цель v1** (смысл: владелец
> видит, что плагин приносит деньги/снимает рутину).

## 17. Склады / origin (точка отправления)

> 📎 **Новый раздел** — выявлен в ревью. Origin (откуда едет посылка) есть у всех, но по-разному:
> - **Яндекс:** полноценный **мульти-склад** — своя таблица в БД (`..._warehouses`), объект `WC_Data`, CRUD через data-store,

## 18. Multi-shipping и кросс-плагинная координация

Основной класс плагинов доставки `Woodev_Shipping_Plugin` должен иметь возможность подключать несколько разных служб доставки —
по аналогии с модулем «Оплата», где регистрируется неограниченное количество шлюзов, необязательно от одного провайдера. Т.е.
в одном `Woodev_Shipping_Plugin` можно объявить сразу и CDEK, и Яндекс, каждый со своей доменной логикой и страницей настроек.
Сейчас ни одного multi-shipping плагина нет — это на будущее (клиенты часто просят плагин-агрегатор).

> 📎 Из ревью кода — **зачатки координации уже есть**, и они глубже, чем «на будущее»:
> - **Почта ↔ CDEK:** `integration-edostavka.php` активно координирует два плагина (уступает Dadata, перехватывает данные
>   локации, мапит `city_code`→`postal_code`, синхронизирует выбор города через общий AJAX).
> - **Машинерия модулей:** у Почты есть `install-module` / `uninstall-module` / `orders-public` (interfaces + DTO), которые
>   **регистрируют плагин в центральном CMS-бэкенде по GUID** (получают `guid_id`/`guid_key`, используемые в т.ч. для подписи
>   webhook’ов). Это уже похоже на каркас агрегатора/реестра.
> - **Яндекс:** `competitor-notices` — детектит соседей (Я.GO, CDEK, Почта) и показывает админ-заметки (конфликт/рекомендация).
>
> **Вывод:** «мульти-шиппинг агрегатор» — не с нуля. Нужно: (1) формализовать реестр служб внутри одного плагина (как шлюзы в
> оплате); (2) поднять ad-hoc координацию (Dadata-владение, маппинг локаций, взаимные уведомления) до общего слоя; (3) решить
> судьбу GUID-регистрации в CMS-бэкенде.

✅ **РЕШЕНО (s32) — вариант (б), оставляем как легаси Почты.** ⚠️ **Поправка к черновику:** формулировка выше
> «это уже похоже на каркас агрегатора/реестра» — **пере-интерпретация** (Claude в сессии структурирования). На
> деле `guid_id`/`guid_key` — чистая доменная интеграция Почты РФ с её бэкендом «Отправка», нигде больше не
> используется и к мульти-доставке отношения не имеет. Данные-контракт сохраняем при миграции, во фреймворк **не
> обобщаем**. Мульти-перевозчик — только **шов реестра** N-перевозчиков (как шлюзы в оплате), без машинерии
> агрегатора наперёд. Координация уже поднята: Dadata→§9, competitor-notices→фреймворк-модуль (s28); маппинг
> локаций (Почта↔CDEK `city_code`→`postal_code`) — домен.

## 19. Контракт данных (meta-ключи) — release-blocking

> 📎 **Новый раздел.** По clean-break-политике проекта (v2): **внутренние API ломать можно, данные установленных сайтов —
> нельзя**. shipping-method ID, instance setting keys, order/session meta keys, REST-неймспейсы, AJAX-экшены, cron-хуки,
> имена опций — **сохраняем байт-в-байт**. Ниже — реальные контракты из кода (фиксируем, чтобы рефакторинг их не потерял):
>
> **CDEK** (префикс `_wc_edostavka_`): `_status`, `_cdek_order_id`, `_tracking_code`, `_chosen_delivery_point`,
> `_customer_location`, `_shipping`, `_waybill_downloaded`, `_barcode_downloaded`; опция `woocommerce_edostavka_settings`;
> transient `woocommerce_edostavka_access_token`; cron `wc_edostavka_orders_update`; method id `edostavka`.
>
> **Почта** (префикс `_wc_russian_post_`): `is_russian_post`, `tracking_number`, `status`, `export_type`, `russian_post_id`,
> `tracking_history`, `rate`; опция `woocommerce_wc_russian_post_settings`; `wc_russian_post_guid_id`/`_guid_key`; cron
> `wc_russian_post_orders_update`; method ids `wc_russian_post` (courier/postal).
>
> **Яндекс** (префикс `_yandex_delivery_`): `_request_id`, `_destination_station_id`, `_destination_station_address`,
> `_destination_interval_from`/`_to`, `_sharing_url`, `_state_status`; опция `woocommerce_yandex_delivery_settings`; таблица
> `..._warehouses`; cron `wc_yandex_orders_update`; method ids `yandex_delivery_express` / `yandex_delivery_other_day`.

> 📎 **Вывод для рефакторинга:** фреймворк стандартизирует **форму** этих данных (выбранный ПВЗ, трек-номер, remote-id, статус,
> история), но **ключи каждого плагина сохраняются как есть** — через per-plugin маппинг в
> `docs-internal/migration/<plugin>-data-preservation-checklist.md` на этапе миграции каждого плагина.

✅ **РЕШЕНО (s32).** Таблицы ключей выше = **стартовый референс**. Стратегия — **per-plugin миграция через
> `Woodev_Lifecycle` `upgrade_to_X_Y_Z()`** с **явным** маппингом {легаси-ключ → каноническое поле фреймворка}
> (только плагин знает свои поля, ничего «наугад»). Фреймворк определяет канонические поля (`chosen_pickup_point`,
> `tracking_number`, `remote_order_id`, `delivery_status`, `status_history`, `customer_location`…). На этап
> миграции каждого плагина (в `docs-internal/migration/<plugin>-data-preservation-checklist.md`): мигрировать meta
> **существующих** заказов (батч через `Woodev_Background_Job_Handler`); миграция **идемпотентна**, безопаснее
> **не удалять** легаси-ключи сразу; сверить, что легаси-ключ **никто не читает снаружи** (webhooks, др. плагины,
> тема, выгрузки).
