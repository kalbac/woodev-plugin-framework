## 0. Context and method

The user asked for an independent second-model audit of the ongoing v2.0 refactor after the a7da0ea payment-gateway regression (28 still-called methods were removed, masked by a blanket PHPStan ignore that had a comment claiming "the methods exist at runtime"). The motivation: catch similar masked-fatal patterns before v2.0 ships to 10+ production plugins.

**Method:**
1. Read all refactor-related docs (`PLANS.md`, `platform-v2-implementation-spec.md`, `platform-v2-strategy-alignment.md`, `platform-v2-dependency-matrix.md`, `platform-v2-next-analysis.md`, `platform-v2-roadmap-reconciliation.md`, ADRs 001–004, `CURRENT-STATE.md`, recent `SESSION-LOG.md`).
2. Read `phpstan.neon` and identify every blanket ignore that could mask a `Call to an undefined method` pattern.
3. Run PHPStan with the suspect ignores removed (in a temporary config) to see what's actually being hidden.
4. Cross-reference every flagged call site with the actual class hierarchy in source.
5. Read `woodev/class-plugin.php`, `woodev/class-woocommerce-plugin.php`, `woodev/class-helper.php`, `woodev/class-framework-resolver.php`, `woodev/bootstrap.php`, and the gateway/shipping/box-packer trees end-to-end.
6. Read the recent SESSION-LOG entries that document the Phase 5 cleanup slices and the a7da0ea fix.
7. Validate the realistic fixtures (`tests/_fixtures/woodev-realistic-*-plugin/`) and their tests against what they claim to prove.

All findings are evidence-based, with `file:line` references, regex proofs (PHPStan output captured in the per-agent task results), and concrete fix sketches.

## 1. Release blockers (must fix before v2.0 release)

### B-1. Three remaining blanket `Call to an undefined method` PHPStan ignores

Same class of bug as a7da0ea. Each pattern is a blanket ignore that hides type errors in code paths called by framework code. **PHPStan with the ignores removed revealed 30 masked errors across 5 patterns.**

#### B-1a. `Woodev_Payment_Gateway_API_Payment_Notification_Response::#` (CLASS-WIDE)

- **Where the ignore is:** `phpstan.neon:120`
- **What it masks:** 6 unguarded calls in `woodev/payment-gateway/class-payment-gateway-hosted.php:440-452` (lines 442, 443, 444, 447, 448, 450). The calls are `$response->get_exp_month()`, `$response->get_exp_year()`, `$response->get_card_type()` (credit-card sub-interface) and `$response->get_loan_type()`, `$response->get_credit_amount()`, `$response->get_first_payment()` (loans sub-interface).
- **The bug:** dispatch is value-based (`switch ( $response->get_payment_type() )`), not type-based. If a hosted gateway plugin returns `PAYMENT_TYPE_CREDIT_CARD` from `get_payment_type()` on a response that does NOT implement `Woodev_Payment_Gateway_API_Payment_Notification_Credit_Card_Response`, line 442 **fatals on checkout** with `Error: Call to undefined method`.
- **Pattern is correct elsewhere:** `woodev/payment-gateway/handlers/abstract-hosted-payment-handler.php:286-296` uses `instanceof` guards correctly. The hosted class just doesn't follow the same pattern.
- **Severity:** **HIGH / P0** — checkout fatals for any hosted gateway plugin that uses a non-standard response type.
- **Fix:** copy the `instanceof` pattern from `abstract-hosted-payment-handler.php:286-296` into `class-payment-gateway-hosted.php:440-452`, then remove the class-wide ignore from `phpstan.neon:120`. (See code in `gotchas/gateway-type-methods-required.md`.)

#### B-1b. `Woodev_Box_Packer_Item::get_product()`

- **Where the ignore is:** `phpstan.neon:127`
- **What it masks:** 1 call in `woodev/box-packer/class-packer-separatly.php:38` — `$item->get_product()` on a typed `Woodev_Box_Packer_Item[]` array.
- **The bug:** the interface `Woodev_Box_Packer_Item` does NOT declare `get_product()`. The method exists only in the framework's own `Woodev_Packer_Item_Implementation`. If a plugin implements `Woodev_Box_Packer_Item` with their own class (the documented extension point), `pack()` on `Woodev_Packer_Separately` **fatals** with `Call to undefined method` on their custom item.
- **Severity:** **HIGH / P0** — broken interface contract. The other 3 packers (`Single_Box`, `Virtual_Box`, `Boxes`) do NOT call `get_product()`, so the risk is narrow but real for any plugin using `Packer_Separately` with custom items.
- **Fix:** split into `Woodev_Box_Packer_Item` (base, no product) + `Woodev_Box_Packer_Item_With_Product extends Woodev_Box_Packer_Item` (adds `get_product()`), and have `Woodev_Packer_Separately` declare it requires the extended type. OR add `get_product()` to the base interface with a no-op default. Then remove the ignore.

#### B-1c. `Woodev\Framework\Shipping\Shipping_API` interface — broken contract

- **Where the ignore is:** `phpstan.neon:130-131` (scoped to the interface file)
- **What it masks:** 20 type errors in 5 methods of the interface. The interface references 6 types that **do not exist in the framework**:
  - `Woodev_Shipping_API_Rate_Response` (return)
  - `Woodev_Shipping_API_Order_Response` (return, 3 sites)
  - `Woodev_Shipping_API_Tracking_Response` (return)
  - `Woodev_Shipping_API_Pickup_Points_Response` (return)
  - `Woodev_Exportable_Order` (parameter)
  - `Woodev_Shipping_Exception` (must extend `Throwable` per `@throws`, 6 sites)
- **The bug:** the namespaced interface was added in v2 (per the file's `@since 1.5.0`) but the concrete response classes and exception class were not ported from any production plugin. The interface is **unimplementable** by a type-safe plugin.
- **Severity:** **HIGH / P0** — broken contract, not a latent fatal. Any plugin that tries to `implements Shipping_API` will be type-unsafe.
- **Fix:** port the 6 types from `plugins-reference/woocommerce-edostavka` (which has the full legacy implementation) into the framework, generalize their docblocks/constructors, namespace them under `Woodev\Framework\Shipping\*`. Then remove the ignore. See `gotchas/shipping-api-broken-contract.md`.

### B-2. `Woodev_Plugin::get_woocommerce_uploads_path()` — WC-specific method leaked into base

- **Where:** `woodev/class-plugin.php:1258-1263`
- **What:** a static method that returns a `/woocommerce_uploads/` path. By definition, this is a WooCommerce API helper. It is publicly documented in `docs/core-framework.md:649` as part of the platform-neutral base class.
- **The bug:** pure-WordPress plugins extending `Woodev_Plugin` would inherit a method that returns a WC-specific path. The method is called via `Woodev_Plugin::get_woocommerce_uploads_path()` or `$plugin->get_woocommerce_uploads_path()`. Moving it breaks the 10+ existing plugins.
- **Severity:** **HIGH** for the v2 split goal. Not a runtime bug today (existing plugins call it and get the right path). The leak is the issue — it contradicts the documented v2 invariant that `Woodev_Plugin` is platform-neutral.
- **Fix:** move the implementation to `Woodev_Woocommerce_Plugin`, keep a deprecation shim on the base that calls `static::class . '::' . 'get_woocommerce_uploads_path'`. The shim must use the concrete-plugin class name (since `static::` doesn't work through base static calls in older PHP). Update `docs/core-framework.md:649`.

### B-3. `Woodev_Plugin::get_blocks_handler()` — non-nullable typed return TypeError trap

- **Where:** `woodev/class-plugin.php:71` (property) and `:1018-1020` (method)
- **What:** property `protected Woodev_Blocks_Handler $blocks_handler;` (typed, no default, non-nullable). Method `public function get_blocks_handler(): Woodev_Blocks_Handler { return $this->blocks_handler; }` (non-nullable return).
- **The bug:** the base constructor never sets `$blocks_handler`. Only `Woodev_Woocommerce_Plugin::init_blocks_handler()` initializes it (called from `__construct` AFTER `parent::__construct()` returns). A pure-WordPress subclass that calls `get_blocks_handler()` triggers `TypeError: Typed property must not be accessed before initialization`.
- **Severity:** **HIGH** for the v2 split goal. Not a runtime bug today (no pure-WP plugin in production yet), but the contract lies to the type system.
- **Fix:** change to `protected ?Woodev_Blocks_Handler $blocks_handler = null;` and `public function get_blocks_handler(): ?Woodev_Blocks_Handler`. Callers that need a non-null handler must guard. See `gotchas/blocks-handler-typed-property-trap.md`.

## 2. High-priority issues (not release-blockers, but blockers for shipping v2.0-stable)

### H1. PHP 8.4+ implicit-nullable deprecations masked by test workaround

- **Where the mask is:** `tests/unit/RealisticPaymentFixtureTest.php:88-94` wraps `load_plugins()` in `error_reporting( error_reporting() & ~E_DEPRECATED )`.
- **What it hides:** legacy payment handler files (`class-payment-gateway*`, `handlers/abstract-*.php`, `handlers/capture.php`, `admin/class-payment-gateway-admin-*.php`) use the PHP 7.4-era implicit-nullable syntax `$arg = null` without `?Type`. This is a hard `E_DEPRECATED` in PHP 8.4 (November 2024) and will be a fatal in PHP 9.0.
- **Why this is a real risk:** the test workaround means `composer check` cannot surface the deprecations. When a customer runs a production plugin on PHP 8.4+, the deprecations will appear in real logs.
- **Fix:** targeted audit of all implicit-nullable parameters in the payment handler file tree (`grep -nE 'function [a-z_]+\(.*\$\w+\s*=\s*null\)' woodev/payment-gateway/`), add `?Type` annotations, then remove the `error_reporting` mask. See `gotchas/php84-implicit-nullable-payment-handlers.md`.

### H2. Resolver has invisible runtime dependency on legacy `Woodev_Plugin_Bootstrap`

- **Where:** `woodev/class-framework-resolver.php:161-162, 210` — three places call `\Woodev_Plugin_Bootstrap::instance()` for notice wiring.
- **The bug:** the "minimal" resolver (per the spec, section 6.1) references the legacy compat class. If a plugin instantiates the resolver directly without loading `bootstrap.php`, the resolver fatals the moment any plugin is incompatible.
- **Why tests don't catch it:** the realistic fixtures use happy-path data with zero incompatible plugins, so `has_update_notices()` returns `false` and these lines never execute. The dependency is invisible to the test suite.
- **Fix:** either (a) inject notice callbacks via constructor args (testable, boundary-clean), or (b) move the `add_action` wiring into `Woodev_Plugin_Bootstrap::load_plugins()` (less code churn, resolver exposes only payloads). Option (a) preferred per the spec's "resolver decides, bootstrap delegates" rule.

### H3. `load_plugins()` is not idempotent

- **Where:** `woodev/class-framework-resolver.php` end of `load_plugins()` method
- **The bug:** second invocation re-loads `class-plugin.php` (no-op via `require_once`) and re-runs every callback, doubling side effects (action firings, registrations, notifications).
- **Risk:** long-running WP-Cron and Action Scheduler processes can call this multiple times. A license check that fires an admin notice twice, or a callback that registers cron schedules, can produce subtle bugs.
- **Fix:** add a `$loaded` guard at the top of `load_plugins()`: `if ( $this->loaded ) { return; } $this->loaded = true;`.

### H4. Resolver does not deduplicate by `plugin_id`

- **Where:** `register_loader_definition()` and `register_legacy_plugin()` in the resolver
- **The bug:** two plugins registered with the same `plugin_id` both end up in `active_plugins`. They will collide on:
  - Option keys (`woodev_{id}_*`)
  - Cron schedules
  - License option keys
  - WC logger handle (`wc_get_logger()->add( 'foo', ... )`)
  - Action Scheduler group names
- **Risk:** if a site accidentally has two copies of the same plugin (staging + production, leftover after a refactor, multisite migration), the second copy silently breaks the first.
- **Fix:** add a `if ( isset( $this->plugin_ids[ $plugin_id ] ) ) { return false; }` guard at the top of registration methods, and record in `invalid_loader_definitions`.

## 3. Medium-priority issues (technical debt to plan against)

### M-1. `Woodev_Helper` retains hard WC coupling in non-fallback paths

- **Where:** `woodev/class-helper.php:486-541, 550-568`
- **What:** three public static methods are unconditionally typed against WC classes:
  - `get_order_line_items( WC_Order $order ): array` (line 490) — type-hints `WC_Order`, `WC_Order_Item_Product`, `WC_Product`
  - `is_order_virtual( WC_Order $order ): bool` (line 550) — same
  - `shop_has_virtual_products(): bool` (line 575) — `function_exists('wc_get_products')` guard OK, but the surrounding code is WC-coupled
- **The 14 Phase 5 cleanup slices missed these** because they were the "no fallback" path — there was no `wc_*` helper call to replace, just direct WC class usage.
- **Fix:** move to a new `Woodev_Woocommerce_Helper` class (file: `woodev/class-woocommerce-helper.php`). Pattern is consistent with the v2 split intent. Should be done as a single helper-class split, NOT as more `function_exists()` fallback slices.

### M-2. `Woodev_Plugin_Bootstrap::is_woocommerce_active()` duplicates `Woodev_Helper::is_woocommerce_active()`

- **Where:** `woodev/bootstrap.php:154-162` vs `woodev/class-helper.php:715-724`
- **Fix:** have the bootstrap version delegate to the helper. Both must stay public and static for backward compat; only the body changes.

### M-3. `Woodev_Plugin` — no-WordPress plugin subclass fails for `get_documentation_url()`, `get_support_url()`, `get_sales_page_url()`

- **Where:** `woodev/class-plugin.php:1187, 1196, 1205`
- **What:** these return `null`/`''` in the base. A pure-WP plugin that uses these would silently get empty values. The contract is "plugins override" — but the docs don't say so.
- **Fix:** add docblock `@since 2.0.0 Must be overridden by plugin subclasses; returns null in base.` to each.

### M-4. `add_class_form_wrap_start()` / `_end()` on `Woodev_Plugin` is WC-specific (admin settings HTML)

- **Where:** `woodev/class-plugin.php:522, 528`
- **What:** outputs `<div class="woodev-licence-need">`. The docblock says it is "invoked by `Woocommerce_Plugin` via WC settings hooks" — i.e., it is hook-installed by the WC subclass. Should be in `Woodev_Woocommerce_Plugin`.
- **Risk:** low. Pattern is correct (subclass installs the hook), but the implementation lives on the base.
- **Fix:** move to `Woodev_Woocommerce_Plugin`. Add a deprecated stub on the base if any external code calls it directly.

### M-5. Indentation/formatting drift in `class-framework-resolver.php`

- **Where:** `woodev/class-framework-resolver.php:435, 543, 564, 567, 568`
- **What:** mixed tabs/spaces. `composer phpcs` currently passes, but the drift is a maintenance hazard.
- **Fix:** `composer phpcbf` to auto-fix, re-run `composer check`. Should be a 1-minute cleanup.

## 4. Lower-priority issues (track for post-v2.0-stable)

### L-1. `Woodev_Plugin::VERSION` and `@version` docblock inconsistency

- `class-plugin.php:15` says `@version 1.4.0`, `:20` says `const VERSION = '1.4.1'`. Cosmetic.

### L-2. Test coverage gaps (per `FrameworkResolverTest` audit)

The spec (section 12.2) requires tests for the following, but the test suite does not cover them:
- `backwards_compatible` window behavior
- Multi-version framework arbitration
- `fails_wordpress_requirement` with the legacy `minimum_wp_version` field
- Bootstrap delegation chain (no test goes through `Woodev_Plugin_Bootstrap::instance()->register_loader_definition()`)
- Resolver boundary negative assertions (spec section 6.4: "Resolver tests must fail if the resolver starts owning runtime platform behavior" — currently only positive tests)

### L-3. `Woodev_Plugin::__construct` calls `add_woocommerce_hooks()` (empty stub)

- `woodev/class-plugin.php:264` — empty `protected function add_woocommerce_hooks(): void {}` called from `add_hooks()`. This is the cleanest possible extension point for `Woodev_Woocommerce_Plugin` overrides. **No issue** — calling out as a positive pattern. Document for posterity in a wiki article.

### L-4. `Woodev_Helper::render_select2_ajax()` is WC-coupled by design

- `woodev/class-helper.php:998-1117` — embeds inline JS that requires WC's `wc_enhanced_select_params` global. JS degrades gracefully, but the action is misleadingly named `woodev_wc_select2_ajax_rendered`. Should move to a WC-only helper class alongside M-1.

### L-5. `class-lifecycle.php:208-220` docblock mentions `WC_Admin_Settings` but code is platform-neutral

- Misleading comment. The implementation just iterates and calls `update_option`. No WC dep. Cosmetic.

### L-6. `get_framework_file()` returns `__FILE__` — correct in multi-version arbitration, but should be documented

- The method always returns the file containing the class definition. In multi-version scenarios, the highest-version framework's `__FILE__` is the active copy. The behavior is correct, but the docblock should mention this.

### L-7. 166 tests / 338 assertions is thin for a framework with 10+ dependent plugins

- Most of the recent work added helper/cleanup tests. End-to-end coverage of a concrete gateway through full WC runtime is missing. The integration test layer (`tests/integration/`) requires `wp-env` and is not part of `composer check`. A new "concrete gateway in WP test framework" smoke test would have caught a7da0ea.

## 5. Observations on the refactor process itself

The user noted "начало вроде было не плохое. [...] Но вот потом, что-то всё как будто не туда пошло." Below is a structured read of what happened, not a critique of any agent, intended for a future session to plan against.

### 5.1 What worked
- **Phase 0–4** (spec acceptance, resolver, loader, platform split, early class availability, specialized bases) is faithful to the implementation spec and genuinely tested in source, not only in docs.
- **Realistic fixtures** (`woodev-realistic-shipping-plugin`, `woodev-realistic-payment-plugin`) are framework-owned, generic, and prove what they claim — they close the "framework not validated against a realistic plugin shape" gap that the roadmap reconciliation flagged.
- **Sandbox discipline** held: no edits to `plugins-reference/`, no Phase 6B, no resolver scope expansion.
- **a7da0ea** was a textbook recovery: root cause analysis, restore, blanket-ignore removal, test extension, dedup with the s3 gotcha.

### 5.2 What went off track (or felt off)
- **Paperwork over code** for ~3 sessions. Multiple ADRs, 3+ reference migration drafts (`platform-v2-phase6a-reference-gap-analysis.md`, `edostavka-reference-contract-draft.md`, `yandex-reference-contract-draft.md`), plus the platform v2 strategy alignment doc and next-analysis doc — a lot of planning artifacts for code that hasn't been touched.
- **Admin/licensing was next on the user's plan** (per the user's stated sequence: clean → split → bootstrap/loader → admin/licensing → UI → shipping) but **zero** admin/licensing work happened. The framework's own admin pages and licensing are still the v1 implementation.
- **14 Phase 5 cleanup slices** were "minimally atomic" — one helper replaced per session. This created a maintenance surface (many small `function_exists()` fallbacks spread across base-owned modules) instead of one well-designed helper-class split. The strategy was honest (each slice was test-first, narrow, low-risk) but the **outcome** is technical debt.
- **The deprecation mask** in `RealisticPaymentFixtureTest.php:88-94` is a tell that the cleanup approach hit a wall: rather than fix the implicit-nullable deprecations in payment handler files, the workaround hides them. This is exactly the pattern that masked a7da0ea (workaround that hides a real bug).

### 5.3 What the next session should plan around
- **Fix the 3 release-blocker PHPStan-ignore masks** (B-1a/b/c above). Each fix is bounded and testable. After all 3 are fixed, enable `reportUnmatchedIgnoredErrors: true` in `phpstan.neon:78` to catch future dead ignores.
- **Fix the 2 base-class contract leaks** (B-2, B-3 above). These are the "finish the v2 split properly" work that the 14 Phase 5 slices did not do.
- **Fix the PHP 8.4+ deprecation** (H1). After that, remove the `error_reporting` mask.
- **Decide: rewrite Woodev_Helper OR continue function_exists fallbacks** for the residual WC coupling (M-1, M-4, L-4). The current trajectory is "more slices" — the user should decide if that's the right path or if a single helper-class split is preferable.
- **Plan the admin/licensing work** that the user has flagged for the next phase. This is the "next on the plan" item that has not been started.
- **Update test coverage** with the missing tests in L-2. Most are 5-10 line test methods. The end-to-end integration test (L-7) needs `wp-env` setup.
- **Address the "rewrite-first" reality.** The roadmap reconciliation correctly said the sandbox copies still use the old framework. The user's intent is to rewrite production plugins. That work belongs in production plugin repos, not in this framework repo. Do not start Phase 6B from this repo.
