## Contract posture

- **`needs_guard: yes`** → `guard-yandex-contracts`, `s1-fixture-yandex` (they assert/protect the
  yandex installed-site contract strings — the only unguarded contract zone S1 actually touches
  with live strings).
- **`needs_guard: human-only`** → `s1-invariants-yandex` (constitution path — operator commits).
- **All framework-build tasks** are `needs_guard: no`. They match contract zones only by
  **path-glob** (`woodev/shipping-method/**` → `shipping_method_id`) or by introducing **new
  forward hooks/AJAX/REST names built from the plugin id** — they break NO existing installed-site
  string. Expect a one-glance human pass at the gate, not a guard. This is by design: the
  framework provides *mechanisms*; plugins supply the *contract strings* at rewrite time.

## Why this decomposition fits the loop

- **File-disjoint** → parallel worktrees never collide (runbook §0; the P4-class-plugin.php
  collision the runbook warns about is avoided — only `s1-p6-plugin-wiring` touches it).
- **Dependency-ordered** → `depends_on` serializes real ordering without overlapping `file_set`s.
- **PVZ-map first** (P1) → the highest-value/highest-risk abstraction is proven before the rest.
- **Validation gate last** (`s1-fixture-yandex`) → mirrors the edostavka pilot; proves the
  abstraction fits the #1 reference plugin (audit §7, D-5).
